# Exposing APIs

Exposing an API is a two-step process. First, you register your API specification so the platform knows about your API. Then, you declare the exposure in a Rover file to configure routing and access control. Both steps are applied in the same way using `roverctl apply`.

:::tip Prerequisites
Your team must be onboarded before you can expose APIs. See [Applications](./applications.mdx) for details on how the platform identifies your team and application.
:::

## Step 1: Register Your API Specification

Before exposing an API, you must register its OpenAPI specification. This creates an **Api** resource in the Control Plane, which serves as the central reference for your API's metadata (title, version, category, and base path).

<Tabs groupId="resource-version" defaultValue="legacy">
  <TabItem value="legacy" label="Legacy">

In the legacy format, the specification file is the OpenAPI document itself. Rover-CTL automatically detects it as an API specification based on the `openapi` or `swagger` field and extracts the name from the base path.

```yaml title="checkout-spec.yaml"
openapi: 3.0.0
info:
  title: Checkout API
  version: 1.0.0
  x-api-category: OTHER
servers:
  - url: http://localhost:8080/checkout/v1
    description: Local server for testing
paths:
  /orders:
    get:
      summary: List orders
      responses:
        '200':
          description: Successful response
```

Apply the specification:

```bash
roverctl apply -f checkout-spec.yaml
```

  </TabItem>
  <TabItem value="v1" label="v1">

In the v1 format, the specification is a dedicated Custom Resource. The OpenAPI document content is uploaded to the File Manager separately, and the `specification` field references the resulting file ID.

```yaml title="checkout-apispec.yaml"
apiVersion: rover.cp.ei.telekom.de/v1
kind: ApiSpecification
metadata:
  name: checkout-v1
  namespace: dev--payment-services--checkout
spec:
  specification: "file-manager-id-abc123"
  basepath: /checkout/v1
  version: "1.0.0"
  hash: "sha256:a1b2c3d4..."
  category: other
```

Apply the specification:

```bash
kubectl apply -f checkout-apispec.yaml
```

  </TabItem>
</Tabs>

### What Happens

The ApiSpecification handler processes your specification and creates an **Api** resource. This resource stores the API's metadata — including its base path, version, and category — and makes the API discoverable in the platform. The specification content is stored in the **File Manager** for later retrieval.

## Step 2: Expose the API in Your Rover

Once the API specification is registered, you can declare the exposure in your Rover file. The `basePath` in your exposure must match the base path registered in Step 1.

<Tabs groupId="resource-version" defaultValue="legacy">
  <TabItem value="legacy" label="Legacy">

```yaml title="rover.yaml"
apiVersion: tcp.ei.telekom.de/v1
kind: Rover
metadata:
  name: checkout-service
spec:
  zone: dataplane1
  exposures:
    - type: api
      visibility: ENTERPRISE
      basePath: /checkout/v1
      upstream: https://checkout.internal:8080
      approval: SIMPLE
```

Apply the Rover file:

```bash
roverctl apply -f rover.yaml
```

  </TabItem>
  <TabItem value="v1" label="v1">

```yaml title="rover.yaml"
apiVersion: rover.cp.ei.telekom.de/v1
kind: Rover
metadata:
  name: checkout-service
  namespace: dev--payment-services--checkout
spec:
  zone: dataplane1
  exposures:
    - api:
        basePath: /checkout/v1
        upstreams:
          - url: https://checkout.internal:8080
            weight: 1
        approval:
          strategy: Simple
```

Apply the Rover file:

```bash
kubectl apply -f rover.yaml
```

  </TabItem>
</Tabs>

### Key Fields

| Field | Description |
| ----- | ----------- |
| `basePath` | The public API path. Must match an existing Api resource and be unique across the platform. This is how consumers will reach your API. |
| `upstream` / `upstreams` | The internal URL where your service is running. The gateway proxies incoming requests to this address. In v1, this is an array of upstream objects with optional weights for load balancing. |
| `approval` | The approval strategy for subscription requests: `auto`, `simple`, or `foureyes`. See [Approvals](./approvals.mdx). |

### What Happens

When you apply the Rover file, the Control Plane:

1. Creates an **Application** resource for your service (if it does not already exist)
2. Creates an **ApiExposure** linking your application to the Api resource
3. The ApiExposure handler verifies that the **Api** resource exists (created in Step 1) and configures a **Gateway Route** that proxies traffic from the base path to your upstream

:::caution
If the Api resource does not exist (i.e., the specification was not registered in Step 1), the ApiExposure will remain in a pending state until the specification is applied.
:::

The API is now discoverable by other teams and ready to receive subscription requests.

## Traffic Management

You can configure traffic management settings for your exposed APIs:

<Tabs groupId="resource-version" defaultValue="legacy">
  <TabItem value="legacy" label="Legacy">

```yaml
exposures:
  - type: api
    visibility: ENTERPRISE
    basePath: /checkout/v1
    upstream: https://checkout.internal:8080
    rateLimit:
      provider:
        second: 100
      consumerDefault:
        second: 10
    loadBalancing:
      servers:
        - upstream: https://checkout.internal:8080
          weight: 1
```

  </TabItem>
  <TabItem value="v1" label="v1">

```yaml
exposures:
  - api:
      basePath: /checkout/v1
      upstreams:
        - url: https://checkout.internal:8080
          weight: 1
      approval:
        strategy: Simple
      traffic:
        rateLimit:
          provider:
            limits:
              second: 100
          consumers:
            default:
              limits:
                second: 10
        loadBalancing:
          strategy: RoundRobin
```

  </TabItem>
</Tabs>
